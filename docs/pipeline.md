# Pipeline

The v0 pipeline is intentionally narrow:

1. Load PDF bytes and compute a document fingerprint.
2. If `--cache-dir` is provided, compute a cache key from parser version, backend name/version, document fingerprint, OCR adapter state, and span-geometry mode.
3. Return a cached artifact on cache hit. If the snapshot cannot be read, decoded, or validated against the requested key/parser/backend/source envelope, ignore it as a cache miss and emit a `cache_snapshot_ignored` warning instead of failing the parse.
4. Select a PDF backend. The only enabled backend today is `lopdf`, but command logic now talks to a backend interface rather than direct `lopdf` calls.
5. Open the document with the selected backend.
6. Enumerate pages in PDF order. Page-debug paths can request one page directly instead of extracting the full document.
7. Extract native text for each selected page.
8. If `--span-geometry` is enabled, collect candidate native spans and approximate bounding boxes in effective page-local coordinates for bounded small/simple positioned text operations, including simple `Tm` text-matrix axes, `q`/`Q`/`cm` content-matrix transforms, text-state parameters preserved across text objects, `TJ`/`Tc`/`Tw`/`Tz` spacing adjustments, `Ts` text rise, `TL` leading for `T*` line movement, and `'`/`"` shortcut text showing.
9. Accept positioned spans only when their decoded text is compatible with the native text output; otherwise keep the existing page-wide span fallback. Without `--span-geometry`, skip this work entirely. If the requested span-geometry pass hits the bounded content/text cap or unsupported geometry such as page rotation, emit `span_geometry_capped` instead of treating page-wide boxes as complete bbox detail. When positioned spans are accepted, summarize bbox overlap with a bounded sorted spatial scan so layout-risk detection does not become a full dedup/render pass.
10. Build cheap page signals:
    - native span count.
    - native text bytes.
    - glyph count.
    - image coverage from unioned drawn image XObject boxes clipped to the effective page box, nested form-wrapped image transforms, and detected inline image operators when available, with an empty-content fallback hint.
    - duplicate character ratio.
    - bbox overlap ratio from accepted positioned spans.
    - broken encoding ratio.
    - page dimensions from the effective `/CropBox` when present, otherwise `/MediaBox`, including values inherited from parent page-tree nodes.
    - rotation, including inherited `/Rotate` values from the page tree.
    - table-like text density and stroked ruling-line density.
    - annotation count from page `/Annots`, plus form field count from catalog `/AcroForm /Fields` and page widget annotations.
    - whether requested span geometry was capped.
11. Classify the page into a route:
    - `native_fast_path`.
    - `needs_fallback`.
    - `ocr_fallback`.
    - `unsupported`.
    The route decision includes stable `reasons` strings so debug output and downstream agents can explain why fallback work was requested.
    Pages with high image coverage and either missing or very sparse native text route to `ocr_fallback` instead of being reported as complete native extraction.
    Pages with high image coverage and broken native encoding also route to `ocr_fallback` with `broken_encoding_with_image_coverage` so bad embedded text is not treated as complete when an image-backed fallback is available.
    Pages with high image coverage and substantial native text skip OCR but route to layout review with `image_text_overlay`.
12. If an OCR adapter is configured and the page routes to `ocr_fallback`, load sidecar text for that page or invoke the configured OCR command/HTTP endpoint, bounded by `--ocr-timeout-ms`. The default `--ocr-command-input pdf-page` contract passes the PDF path and zero-based page index. With `--backend pdfium --ocr-command-input rendered-image`, Glyphrush renders only OCR-routed pages to temporary PPM files, records render timing, and removes the temporary file after OCR returns. Command OCR receives the rendered image path and page index; HTTP OCR receives `rendered_image_path` and `page_index` JSON instead of `pdf_path`. Non-rendering backends reject the rendered-image contract when a routed page would require it instead of silently switching contracts. Use `ocr-check <pdf> --page-index <N> --strict` to preflight the configured adapter before scanned/hybrid benchmark runs, including the PDFium rendered-image command or HTTP contract when `--backend pdfium --ocr-command-input rendered-image` is supplied.
13. Split native or OCR text into deterministic layout blocks, with conservative reflow for short extraction fragments. If OCR text is applied for an OCR-routed page, derive layout and text views from the OCR text while preserving native spans separately for provenance. When accepted native span geometry exists, first assemble same-baseline positioned fragments into readable text rows so character-level PDFium segments do not lower text recall before layout; overlapping fragments contribute only their non-duplicated suffix so `Vo` + overlapping `Voltage` becomes `Voltage` instead of `VoVoltage`. When table recovery is routed, group aligned multi-column rows into a table block, preserve omitted blank cells against the inferred column grid, merge same-line fragmented positioned cells and near-adjacent single- or multi-cell wrapped positioned continuations into the same row, keep aligned whitespace, cross-column positioned interior section rows, first-column interior positioned section rows, or fragmented first-column interior positioned section rows inside the table grid with blank cells for remaining columns, and keep top/bottom positioned captions as surrounding text blocks. Positioned row runs that are actually bullet lists, including marker-only rows followed by indented text, are normalized as list blocks before table acceptance so aligned bullets do not become false table artifacts. Otherwise keep full-width title/banner bands, leading, middle, and trailing cross-column bands, conservative short section-heading separators, and clearly separated 2-4 column reading order, or group spans by vertical gaps, and assign each layout block the union box of its contributing spans. If table recovery was routed, preserve simple whitespace rows with a consistent column count as table blocks; fixed-width whitespace rows align to header column starts, keep blank cells, and merge lowercase wrapped descriptor fragments into the following row; header-guided whitespace rows can merge same-line or lowercase wrapped leading multi-word descriptor cells against short header columns, merge lowercase trailing descriptor continuations into the preceding row, keep inferred trailing blank cells explicit, preserve section rows, and keep leading captions outside the structured grid; pipe/tab table rows preserve empty delimited cells so blank columns are not collapsed; markdown separator rows such as `| --- | --- |` are ignored as structure, not data. After page-local layout, classify repeated positioned blocks in the top or bottom page margin as headers or footers when the same normalized text appears on multiple pages.
14. Emit a deterministic document artifact with parser/backend/source metadata, page fingerprints over output-relevant text/signals/geometry, quality flags, and separate OCR-required/OCR-applied diagnostics.
15. Store the artifact on cache miss when `--cache-dir` is enabled.

`glyphrush backend-check` is the current backend preflight surface. The global backend choice defaults to `auto`, so plain builds select `lopdf` and PDFium-feature builds select `pdfium` unless `--backend lopdf` or `--backend pdfium` is specified explicitly. `backend-check` emits `glyphrush-backend-check-report-v1` with parser version, selected backend, enabled backend count, candidate backend count, and per-backend capability/limitation metadata. Add `--pdf <file-or-directory>` to smoke the selected backend through load, page counting, native extraction, and classifier/artifact summarization without OCR. Single-file reports include success, wall time, source size, document fingerprint, page counts, native text bytes, image artifact count, fallback pages, OCR-required pages, and stable failure `error_kind` values such as `encrypted_pdf_requires_password` when the backend can identify an unsupported input class before extraction. Directory reports scan top-level `.pdf`/`.PDF` files in stable filename order, aggregate the same counters, include sorted per-document smoke results, and accept `--jobs <N>` for concurrent per-PDF smoke runs with deterministic merge order. PDFium reuses the loaded document handle for page counting and extraction, but caps corpus-level document workers to one to avoid concurrent live PDFium document extraction; directory reports make that visible through `worker_count`. Default builds report `lopdf` as enabled and PDFium/MuPDF as `not_wired`; PDFium-feature builds report `lopdf` and `pdfium` as enabled, with `render_pages: true` for page-image OCR handoff and `builtin_ocr: false` because OCR remains adapter-based.

`glyphrush feature-parity` is the LiteParse comparison surface. It emits `glyphrush-feature-parity-report-v1` with parser/backend provenance, a conservative capability matrix, summary counts for implemented, partial, planned, and intentionally-not-planned items, and a `readiness` block that separates native-text speed-race readiness from full LiteParse drop-in parity. Capability status can depend on the selected backend: PDFium-feature runs report rendered-page OCR handoff as implemented because PDFium can render OCR-routed pages, while non-rendering backends keep that capability partial and explicit. Use it next to `bench --require-speedup-claim liteparse=... --require-speedup-claim liteparse-no-ocr=...`: parity explains what feature gaps remain, while the benchmark gate proves whether default OCR-capable and native-text-only speed claims are quality-backed on a labeled corpus. Add `bench --require-coverage-preset glyphrush-v0` when the benchmark job itself should fail unless the embedded eval report covers the core v0 corpus classes. Add `feature-parity --bench-report <saved-bench-json> --require-speed-evidence` when the parity command itself should fail unless that saved benchmark report contains passing, quality-backed `liteparse` and `liteparse-no-ocr` speedup claims. When the saved report embeds eval category summaries, feature parity includes `benchmark_evidence.quality_categories` so the speed claim carries visible corpus-class coverage instead of only aggregate pass/fail evidence; add `--require-coverage-preset glyphrush-v0` there to make the parity command fail until the saved benchmark covers the same core v0 classes.

The current CLI does not bundle Tesseract or make hidden network OCR calls. It supports explicit sidecar, external-command, HTTP OCR, and local Tesseract rendered-image wrapper adapters so fallback merge behavior is testable without putting OCR in the default hot path. HTTP OCR accepts plain response bodies as OCR text and `application/json` responses with a string `text` field. `ocr-check` reports adapter readiness with non-empty output status, digest/text counts, timeout state, stderr preview, and stable failure kinds such as `missing_dependency` or `http_response_decode_failed`, which makes missing Tesseract language data or malformed OCR-service responses visible before a LiteParse-style OCR comparison is treated as a parser result. Layout reconstruction is currently text-derived with limited fragment reflow, full-width band and short section-separator preservation before conservative 2-4 column ordering for positioned native spans, conservative table-route whitespace/header-guided rows or aligned-positioned row-run recovery that preserves surrounding text blocks, optional structured table rows/cells, and optional geometry-aware block/cell boxes. The `lopdf` backend can emit approximate page-local boxes for bounded simple text-positioning operators when `--span-geometry` is requested, while the `pdfium` backend can emit PDFium text-segment boxes behind the same opt-in flag. The lightweight `lopdf` path estimates direct, form-wrapped, and inline image coverage for page classification, including skipped inline image operators when bytes or colorspace are unsupported by the lightweight decoder. Image artifacts are clipped against the effective page box before page-level coverage is scored, so non-zero `/CropBox` pages do not undercount visible scan/image coverage. Richer geometry-aware layout and advanced table reconstruction are later milestones.
